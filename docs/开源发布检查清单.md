# 开源发布检查清单

把项目发布到 GitHub 前，建议按这个清单检查一次。

## 1. 密钥与敏感信息

- `.env` 必须保留在本地，不提交。
- `.env.example` 只能写占位值，例如 `your_api_key`。
- 已经在聊天、截图或日志中暴露过的 API Key，建议到服务商后台轮换。
- 不要把 `docker compose config` 的输出贴到公开位置，因为它会展开 `.env` 中的真实环境变量。

推荐检查命令：

```powershell
rg "sk-|api[_-]?key|secret|token" -g "!.env" -g "!.venv/**" -g "!__pycache__/**"
python scripts/open_source_audit.py
```

## 2. 论文与数据授权

- `data/raw_docs/*.md` 是演示用样例笔记，可以提交。
- `data/raw_docs/*.pdf` 默认忽略，不建议提交真实论文 PDF。
- 如果确实要提交 PDF，先确认论文许可允许再分发。
- 实验记录如果包含真实项目、姓名、单位、未公开指标，需要脱敏后再提交。

## 3. 生成物与缓存

以下内容不建议提交：

- `.venv/`
- `.pytest_cache/`
- `__pycache__/`
- `vector_store/faiss_index/*`
- `data/processed_docs/*`
- `data/eval/eval_result.csv`

## 4. 发布前验证

```powershell
.\.venv\Scripts\python.exe -m pytest
ruff check .
python scripts/open_source_audit.py
docker compose config --quiet
```

有 API Key 时可以额外验证：

```powershell
python scripts/build_index.py
python scripts/run_eval.py --limit 5
uvicorn app.main:app --host 0.0.0.0 --port 8010 --reload
```

## 5. README 检查

- 项目定位是否一句话能讲清楚。
- 是否有启动命令、API 入口、Swagger 地址和前端地址。
- 是否解释了 RAG、Agent、索引、chunk、评测指标。
- 是否说明样例 Markdown 与真实 PDF 的区别。
- 是否说明当前局限和后续优化方向。
- 是否包含截图、3 分钟体验路径、工程质量检查和许可证说明。

## 6. GitHub 门面

- `LICENSE` 已存在，许可证与 README 一致。
- `CONTRIBUTING.md`、`SECURITY.md` 和 `CODE_OF_CONDUCT.md` 已存在。
- `.github/ISSUE_TEMPLATE/` 和 PR 模板已存在。
- GitHub Actions 至少覆盖 audit、lint、test 和 Docker build。
- Dependabot 已配置，用于依赖和 GitHub Actions 更新提醒。
