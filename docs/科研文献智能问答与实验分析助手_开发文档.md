# 1. 项目背景

在大模型应用开发中，RAG、Agent、工具调用、后端接口封装和效果评估是非常常见的岗位要求。  
单纯调用大模型 API 生成回答，很难体现工程能力；单纯做爬虫或普通 Web 系统，也不够贴合大模型应用开发岗位。

本项目选择“科研文献智能问答与实验分析助手”作为场景，原因如下：

1. 场景足够自然：科研文献、论文笔记、实验记录都属于典型非结构化和半结构化数据。
2. 与个人背景匹配：如果简历中已有论文阅读、模型评估、知识蒸馏、LoRA 微调、实验记录整理等经历，该项目更容易形成闭环。
3. 技术点完整：可以覆盖文档解析、文本切分、向量检索、RAG 问答、多文档对比、Agent 路由、工具调用、评测优化等内容。
4. 面试可讲性强：面试官可以从 RAG、Agent、Prompt、检索效果、评测指标、后端接口等角度继续追问。
5. 实现难度适中：不需要复杂前端和真实业务数据库，MVP 版本可以在 1 到 2 周内完成。

# 2. 项目名称与定位

## 2.1 项目名称

科研文献智能问答与实验分析助手

## 2.2 项目一句话介绍

本项目基于 LangChain、LangGraph、FastAPI、FAISS 和 DeepSeek API，实现了一个面向科研文献阅读和实验记录分析的 Agentic RAG 系统，支持论文问答、多篇论文对比、实验记录查询和阅读笔记生成。

## 2.3 项目定位

本项目不是普通聊天机器人，而是一个完整的大模型应用开发项目，主要体现以下能力：

- RAG 知识库问答开发能力；
- 文献 PDF / Markdown / TXT 解析能力；
- 向量检索和 Embedding 使用能力；
- LangGraph Agent 工作流编排能力；
- FastAPI 后端服务开发能力；
- Prompt 设计与优化能力；
- 基础评测和错误样例分析能力；
- 项目工程组织和 GitHub 展示能力。

# 3. 项目目标

## 3.1 功能目标

系统需要支持以下功能：

1. 上传科研论文、论文笔记和实验记录。
2. 对文档进行解析、清洗、切分和向量化。
3. 使用 FAISS 构建本地向量知识库。
4. 支持围绕论文内容进行自然语言问答。
5. 回答中返回引用来源、页码或文档片段。
6. 支持多篇论文对比，例如方法对比、实验设置对比、优缺点对比。
7. 支持实验记录查询，例如查询某个实验编号、数据集、模型、指标。
8. 支持生成结构化论文阅读笔记。
9. 使用 LangGraph 对不同问题类型进行路由。
10. 使用 FastAPI 封装可调用接口。
11. 构建测试问题集，评估 RAG 问答效果。

## 3.2 简历目标

项目完成后，简历中可以体现：

- 基于 LangChain 的 RAG 应用开发；
- 基于 LangGraph 的 Agent 工作流；
- 基于 FAISS 的向量知识库；
- DeepSeek API 接入；
- FastAPI 接口封装；
- PDF / Markdown 文档解析；
- 多文档问答与引用来源返回；
- RAG 效果评估和参数优化。

## 3.3 MVP 目标

MVP 阶段不追求复杂前端和线上部署，只需要完成核心闭环：

```text
上传文档
  -> 文档解析
  -> 文本切分
  -> 向量化
  -> FAISS 入库
  -> 用户提问
  -> 检索相关片段
  -> LLM 生成答案
  -> 返回引用来源
```

再进一步加入 Agent：

```text
用户提问
  -> 意图分类
  -> 文献问答 / 论文对比 / 实验查询 / 笔记生成
  -> 工具调用
  -> 答案生成
```

# 4. 核心功能范围

## 4.1 必做功能

| 功能 | 说明 | 简历价值 |
|---|---|---|
| 文档上传 | 上传 PDF / Markdown / TXT | 体现文档处理能力 |
| 文本切分 | chunk_size、chunk_overlap | 体现 RAG 基础能力 |
| 向量化 | Embedding 生成 | 体现向量检索能力 |
| FAISS 检索 | TopK 相似片段检索 | 体现知识库能力 |
| RAG 问答 | 基于检索内容生成答案 | 核心能力 |
| 引用来源 | 返回文档名、页码、片段 | 降低幻觉 |
| FastAPI 接口 | 对外提供服务 | 工程能力 |
| LangGraph 路由 | 根据意图调用不同工具 | Agent 能力 |
| 测试集评估 | 命中率、关键词覆盖、耗时 | 优化能力 |

## 4.2 建议功能

| 功能 | 说明 |
|---|---|
| 多篇论文对比 | 对比方法、实验设置、优缺点 |
| 阅读笔记生成 | 自动生成背景、方法、实验、结论 |
| 实验记录查询 | 查询模型、数据集、指标、实验结论 |
| 对话历史 | 支持上下文问答 |
| 流式输出 | 提升体验 |
| LangSmith 追踪 | 方便 debug 和展示工程能力 |

## 4.3 暂不做功能

MVP 阶段不要做以下功能：

1. 登录注册。
2. 权限系统。
3. 多用户隔离。
4. 复杂前端页面。
5. 在线 PDF 高亮。
6. 真实数据库部署。
7. 云端向量库。
8. 微调模型。
9. 分布式任务队列。
10. 复杂知识图谱。

这些功能会拉长周期，不利于快速完成可展示项目。

# 5. 技术选型

## 5.1 Python

建议版本：

```text
Python 3.10 或 Python 3.11
```

原因：

- LangChain、LangGraph、FastAPI、FAISS 兼容性较好；
- Python 生态适合大模型应用开发；
- 面试和实习岗位中更常见。

## 5.2 FastAPI

用途：

- 提供文档上传接口；
- 提供知识库构建接口；
- 提供 RAG 问答接口；
- 提供 Agent 问答接口；
- 提供实验记录查询接口；
- 自动生成 Swagger API 文档。

## 5.3 LangChain

用途：

- 文档加载；
- 文本切分；
- 向量库接入；
- Retriever 检索；
- Prompt 管理；
- RAG Chain 构建。

## 5.4 LangGraph

用途：

- 定义 Agent 状态；
- 定义节点；
- 定义条件路由；
- 组织文献问答、论文对比、实验查询和笔记生成流程。

## 5.5 FAISS

用途：

- 保存文献 chunk 向量；
- 进行相似度检索；
- 本地持久化向量索引。

## 5.6 DeepSeek API

用途：

- RAG 答案生成；
- 问题意图分类；
- 多篇论文对比总结；
- 阅读笔记生成；
- 异常兜底回答。

## 5.7 SQLite

用途：

- 保存文档元信息；
- 保存实验记录；
- 保存问答日志；
- 保存评测结果。

MVP 版本也可以先用 JSON 文件模拟实验记录，后续再换成 SQLite。

## 5.8 Pytest

用途：

- 测试文档解析；
- 测试检索结果；
- 测试 API 接口；
- 测试 Agent 路由。

# 6. 系统总体架构

## 6.1 架构图

```text
用户 / Swagger / Postman
        |
        v
FastAPI 接口层
        |
        v
LangGraph Agent 工作流层
        |
        |-------------------------------|
        |                               |
        v                               v
文献 RAG 工具                    实验记录查询工具
        |                               |
        v                               v
LangChain Retriever              SQLite / JSON
        |
        v
FAISS 向量库
        |
        v
检索文档片段
        |
        v
DeepSeek API 答案生成
        |
        v
返回答案 + 引用来源 + 工具调用结果
```

## 6.2 分层说明

### 6.2.1 API 层

负责接收 HTTP 请求和返回 JSON 响应。

主要接口：

- `/health`
- `/documents/upload`
- `/knowledge/build`
- `/knowledge/search`
- `/chat/rag`
- `/chat/agent`
- `/experiments/search`
- `/eval/run`

### 6.2.2 应用服务层

负责协调核心业务逻辑。

例如：

- 文档上传后保存到本地；
- 调用文档解析模块；
- 调用向量库构建模块；
- 调用 RAG Chain；
- 调用 Agent 工作流。

### 6.2.3 RAG 层

负责完成文献问答：

```text
用户问题 -> Retriever 检索 -> Prompt 拼接 -> LLM 生成 -> 引用返回
```

### 6.2.4 Agent 层

负责判断问题类型并路由：

- literature_qa：文献问答；
- paper_compare：论文对比；
- experiment_query：实验记录查询；
- note_generation：阅读笔记生成；
- general_chat：普通问答。

### 6.2.5 数据层

负责保存数据：

- 本地文档；
- FAISS index；
- 实验记录 JSON / SQLite；
- 问答日志；
- 评测结果。

# 7. 核心业务流程

## 7.1 文档入库流程

```text
上传论文 PDF / Markdown / TXT
        |
        v
保存到 data/raw_docs
        |
        v
文档解析
        |
        v
文本清洗
        |
        v
文本切分
        |
        v
生成 metadata
        |
        v
调用 Embedding
        |
        v
写入 FAISS
        |
        v
保存向量索引
```

## 7.2 RAG 问答流程

```text
用户问题
        |
        v
加载 FAISS 向量库
        |
        v
相似度检索 TopK 文档片段
        |
        v
组装 Prompt
        |
        v
调用 DeepSeek API
        |
        v
生成回答
        |
        v
返回 answer、sources、related_chunks
```

## 7.3 Agent 问答流程

```text
用户问题
        |
        v
classify_intent 节点
        |
        |-- literature_qa
        |       -> retrieve_docs
        |       -> generate_answer
        |
        |-- paper_compare
        |       -> retrieve_docs
        |       -> compare_papers
        |       -> generate_answer
        |
        |-- experiment_query
        |       -> query_experiment_records
        |       -> generate_answer
        |
        |-- note_generation
        |       -> retrieve_docs
        |       -> generate_note
        |
        |-- general_chat
                -> general_answer
```

# 10. 数据设计

## 10.1 文档数据

系统支持三类输入文档：

1. 论文 PDF；
2. Markdown 论文笔记；
3. TXT 技术文档或实验说明。

建议一开始准备以下样例文档：

```text
data/raw_docs/
├── knowledge_distillation_survey.md
├── lora_finetuning_notes.md
└── rag_for_llm_applications.md
```

## 10.2 Chunk 数据结构

文档切分后，每个 chunk 需要保存 metadata。

```json
{
  "chunk_id": "doc_001_chunk_0001",
  "doc_id": "doc_001",
  "doc_name": "knowledge_distillation_survey.md",
  "doc_type": "paper_note",
  "source_path": "data/raw_docs/knowledge_distillation_survey.md",
  "page": 1,
  "section": "Method",
  "text": "知识蒸馏通常通过教师模型的软标签指导学生模型学习..."
}
```

## 10.3 文档元信息表

如果使用 SQLite，可以设计 `documents` 表：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 主键 |
| doc_id | str | 文档编号 |
| filename | str | 文件名 |
| file_type | str | pdf / md / txt |
| path | str | 文件路径 |
| upload_time | datetime | 上传时间 |
| chunk_count | int | 切分后的 chunk 数 |
| status | str | uploaded / indexed |

MVP 阶段也可以先不用数据库，只保存在 metadata 里。

## 11.3 文档加载模块

文件：

```text
app/retriever/loaders.py
```

职责：

- 根据文件类型选择加载方式；
- PDF 使用 `PyPDFLoader`；
- Markdown / TXT 使用普通文本读取；
- 返回 LangChain Document 列表。

设计要点：

1. PDF 需要保留页码；
2. Markdown 需要保留标题信息；
3. TXT 需要保留文件名；
4. 每个 Document 必须有 metadata。

示例代码：

```python
from pathlib import Path
from langchain_core.documents import Document
from langchain_community.document_loaders import PyPDFLoader

def load_single_document(file_path: str) -> list[Document]:
    path = Path(file_path)
    suffix = path.suffix.lower()

    if suffix == ".pdf":
        loader = PyPDFLoader(str(path))
        docs = loader.load()
        for d in docs:
            d.metadata["doc_name"] = path.name
            d.metadata["source_path"] = str(path)
        return docs

    if suffix in [".md", ".txt"]:
        text = path.read_text(encoding="utf-8")
        return [
            Document(
                page_content=text,
                metadata={
                    "doc_name": path.name,
                    "source_path": str(path),
                    "file_type": suffix.replace(".", "")
                }
            )
        ]

    raise ValueError(f"Unsupported file type: {suffix}")


def load_documents_from_dir(docs_dir: str) -> list[Document]:
    docs = []
    for path in Path(docs_dir).glob("*"):
        if path.suffix.lower() in [".pdf", ".md", ".txt"]:
            docs.extend(load_single_document(str(path)))
    return docs
```

## 11.4 文本切分模块

文件：

```text
app/retriever/splitters.py
```

职责：

- 对 Document 进行 chunk 切分；
- 添加 chunk_id；
- 保留 source、page、doc_name 等信息。

示例代码：

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

def split_documents(docs, chunk_size=800, chunk_overlap=120):
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
        separators=["\n\n", "\n", "。", "，", " ", ""]
    )

    chunks = splitter.split_documents(docs)

    for idx, chunk in enumerate(chunks):
        chunk.metadata["chunk_id"] = f"chunk_{idx:05d}"

    return chunks
```

说明：

- `chunk_size` 不宜太小，否则语义不完整；
- `chunk_size` 不宜太大，否则检索不精确；
- 中文文档建议 separators 中加入中文句号和逗号。

示例代码：

```python
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from app.config import settings

def get_embeddings():
    return OpenAIEmbeddings(
        model=settings.embedding_model,
        api_key=settings.deepseek_api_key,
        base_url=settings.deepseek_base_url
    )

def build_faiss_index(chunks):
    embeddings = get_embeddings()
    vector_store = FAISS.from_documents(chunks, embeddings)
    vector_store.save_local(settings.vector_store_path)
    return vector_store

def load_faiss_index():
    embeddings = get_embeddings()
    vector_store = FAISS.load_local(
        settings.vector_store_path,
        embeddings,
        allow_dangerous_deserialization=True
    )
    return vector_store

def get_retriever(top_k=4):
    vector_store = load_faiss_index()
    return vector_store.as_retriever(
        search_kwargs={"k": top_k}
    )
```

```python
from app.retriever.retrieval import search_knowledge_base
from app.chains.llm import get_chat_model
from app.chains.prompts import RAG_PROMPT

def build_context(results):
    context_parts = []
    for i, item in enumerate(results, start=1):
        source = item.get("source", "")
        page = item.get("page", "")
        content = item.get("content", "")
        context_parts.append(
            f"[{i}] 来源：{source}，页码：{page}\n内容：{content}"
        )
    return "\n\n".join(context_parts)

def answer_with_rag(question: str, top_k: int = 4):
    results = search_knowledge_base(question, top_k=top_k)
    context = build_context(results)

    llm = get_chat_model()
    prompt = RAG_PROMPT.format(question=question, context=context)

    response = llm.invoke(prompt)

    return {
        "answer": response.content,
        "sources": list({r["source"] for r in results if r.get("source")}),
        "related_chunks": results
    }
```

# 12. LangGraph Agent 工作流设计

## 12.1 为什么使用 LangGraph

普通 RAG Chain 适合单一问答流程，但本项目有多种任务：

- 文献问答；
- 论文对比；
- 实验记录查询；
- 阅读笔记生成；
- 普通聊天。

这些任务路径不同，因此用 LangGraph 组织成“状态 + 节点 + 条件路由”的形式更清晰。

## 12.2 AgentState 设计

文件：

```text
app/graph/state.py
```

示例代码：

```python
from typing import TypedDict, Optional, List, Dict, Any

class AgentState(TypedDict):
    question: str
    intent: str
    confidence: float

    retrieved_docs: List[Dict[str, Any]]
    experiment_results: List[Dict[str, Any]]

    answer: str
    sources: List[str]
    error: Optional[str]
```

## 12.3 意图类型

| intent | 说明 | 示例 |
|---|---|---|
| literature_qa | 文献知识问答 | 知识蒸馏是什么？ |
| paper_compare | 多篇论文对比 | 对比 LoRA 和知识蒸馏 |
| experiment_query | 实验记录查询 | EXP-002 的准确率是多少？ |
| note_generation | 阅读笔记生成 | 帮我生成这篇论文的阅读笔记 |

| 节点 | 功能 |
|---|---|
| classify_intent | 判断问题意图 |

| compare_papers | 生成论文对比 |

## 12.5 意图分类规则

MVP 阶段建议先用规则，保证可控。

```python
def classify_intent_rule(question: str) -> str:
    q = question.lower()

    if "exp-" in q or "实验" in q or "accuracy" in q or "f1" in q:
        return "experiment_query"

    if "对比" in q or "比较" in q or "区别" in q or "相同点" in q:
        return "paper_compare"

    if "阅读笔记" in q or "总结这篇" in q or "生成笔记" in q:
        return "note_generation"

    if "你好" in q or "hello" in q:
        return "general_chat"

    return "literature_qa"
```

后续可以改成 LLM 分类，让输出为 JSON：

```json

  "confidence": 0.86,
  "reason": "用户要求比较两个技术路线。"
}
```

## 12.6 LangGraph 工作流代码骨架

文件：

```text
app/graph/workflow.py
```

示例代码：

```python
from langgraph.graph import StateGraph, END
from app.graph.state import AgentState
from app.graph.nodes import (
    classify_intent_node,
    retrieve_docs_node,
    query_experiments_node,
    compare_papers_node,
    generate_note_node,
    generate_answer_node,
    general_chat_node,
)

def route_by_intent(state: AgentState):
    intent = state.get("intent")

    if intent == "literature_qa":
        return "retrieve_docs"
    if intent == "paper_compare":
        return "retrieve_docs"
    if intent == "experiment_query":
        return "query_experiments"
    if intent == "note_generation":
        return "retrieve_docs"
    return "general_chat"

def route_after_retrieval(state: AgentState):
    intent = state.get("intent")

    if intent == "paper_compare":
        return "compare_papers"
    if intent == "note_generation":
        return "generate_note"
    return "generate_answer"

def build_agent_workflow():
    graph = StateGraph(AgentState)

    graph.add_node("classify_intent", classify_intent_node)
    graph.add_node("retrieve_docs", retrieve_docs_node)
    graph.add_node("query_experiments", query_experiments_node)
    graph.add_node("compare_papers", compare_papers_node)
    graph.add_node("generate_note", generate_note_node)
    graph.add_node("generate_answer", generate_answer_node)
    graph.add_node("general_chat", general_chat_node)

    graph.set_entry_point("classify_intent")

    graph.add_conditional_edges(
        "classify_intent",
        route_by_intent,
        {
            "retrieve_docs": "retrieve_docs",
            "query_experiments": "query_experiments",
            "general_chat": "general_chat",
        }
    )

    graph.add_conditional_edges(
        "retrieve_docs",
        route_after_retrieval,
        {
            "generate_answer": "generate_answer",
            "compare_papers": "compare_papers",
            "generate_note": "generate_note",
        }
    )

    graph.add_edge("query_experiments", "generate_answer")
    graph.add_edge("compare_papers", END)
    graph.add_edge("generate_note", END)
    graph.add_edge("generate_answer", END)
    graph.add_edge("general_chat", END)

    return graph.compile()
```

# 15. 评测体系设计

## 15.1 为什么要做评测

简历中只写“搭建 RAG 系统”比较普通，如果能写：

- 构建测试问题集；
- 统计引用命中率；
- 统计关键词覆盖率；
- 根据失败样例优化 TopK、chunk size、Prompt；

就会更像真正做过工程优化。

## 15.2 评测数据格式

`data/eval/rag_eval_questions.csv`：

```csv
id,question,expected_source,expected_keywords,intent
1,知识蒸馏的核心思想是什么,knowledge_distillation_survey.md,教师模型;学生模型;软标签,literature_qa
2,LoRA 为什么能减少训练参数,lora_finetuning_notes.md,低秩;参数量;显存,literature_qa
3,对比 LoRA 和知识蒸馏的作用,lora_finetuning_notes.md|knowledge_distillation_survey.md,微调;压缩;学生模型,paper_compare
```

## 15.3 评测指标

| 指标 | 说明 |
|---|---|
| source_hit_rate | 检索结果是否包含预期文档 |
| keyword_hit_rate | 回答是否包含预期关键词 |
| citation_rate | 回答是否返回引用来源 |
| avg_latency | 平均响应时间 |
| failed_case_count | 失败问题数量 |

## 15.4 评测脚本设计

文件：

```text
scripts/run_eval.py
```

流程：

```text
读取 rag_eval_questions.csv
        |
        v
逐条调用 /chat/agent 或内部函数
        |
        v
检查 source 是否命中
        |
        v
检查 answer 是否包含关键词
        |
        v
统计耗时
        |
        v
输出评测报告
```

示例代码：

```python
import time
import pandas as pd
from app.services.agent_service import run_agent

def keyword_hit(answer: str, expected_keywords: str) -> bool:
    keywords = expected_keywords.split(";")
    return any(k in answer for k in keywords)

def source_hit(sources: list[str], expected_source: str) -> bool:
    expected_sources = expected_source.split("|")
    return any(src in sources for src in expected_sources)

def run_eval():
    df = pd.read_csv("data/eval/rag_eval_questions.csv")

    total = len(df)
    source_hits = 0
    keyword_hits = 0
    latencies = []
    failed_cases = []

    for _, row in df.iterrows():
        start = time.time()
        result = run_agent(row["question"])
        latency = time.time() - start
        latencies.append(latency)

        answer = result.get("answer", "")
        sources = result.get("sources", [])

        s_hit = source_hit(sources, row["expected_source"])
        k_hit = keyword_hit(answer, row["expected_keywords"])

        source_hits += int(s_hit)
        keyword_hits += int(k_hit)

        if not s_hit or not k_hit:
            failed_cases.append({
                "id": int(row["id"]),
                "question": row["question"],
                "source_hit": s_hit,
                "keyword_hit": k_hit,
                "answer": answer[:200]
            })

    report = {
        "total": total,
        "source_hit_rate": source_hits / total,
        "keyword_hit_rate": keyword_hits / total,
        "avg_latency": sum(latencies) / total,
        "failed_cases": failed_cases
    }

    print(report)

if __name__ == "__main__":
    run_eval()
```

## 15.5 调优方向

当 source_hit_rate 低：

- 调大 TopK；
- 调小 chunk_size；
- 增加 chunk_overlap；
- 优化文档标题和 metadata；
- 尝试混合检索；
- 增加 rerank。

当 keyword_hit_rate 低：

- 优化 Prompt；
- 增加回答格式要求；
- 检查检索片段是否足够；
- 检查测试问题是否过于宽泛。

当 avg_latency 高：

- 降低 TopK；
- 减少 Prompt 中 context 长度；
- 控制回答长度；
- 开启缓存；
- 对文档提前离线入库。

# 20. 面试讲解思路

## 20.1 1 分钟版本

这个项目是我做的一个科研文献智能问答与实验分析助手，主要用于解决论文阅读和实验记录检索效率低的问题。系统使用 LangChain 完成文档解析、文本切分、向量化和 RAG 问答，使用 FAISS 保存本地向量索引，使用 DeepSeek API 生成回答，并通过 FastAPI 提供接口。除此之外，我还用 LangGraph 做了一个简单 Agent 工作流，可以根据用户问题路由到文献问答、论文对比、实验记录查询或阅读笔记生成。最后我构建了一批测试问题，从引用命中率、关键词覆盖率和响应耗时几个维度评估系统效果。

## 20.2 3 分钟版本

项目主要分为三部分。

第一部分是 RAG 知识库构建。我准备了一批科研文献和实验笔记，先通过文档加载器读取 PDF、Markdown 和 TXT 文件，再用 RecursiveCharacterTextSplitter 进行文本切分，并为每个 chunk 保存文档名、页码、chunk_id 等 metadata。之后调用 Embedding 模型生成向量，使用 FAISS 保存到本地。

第二部分是问答和 Agent 工作流。用户提问后，普通 RAG 流程会先从 FAISS 中检索 TopK 相关片段，再把问题和上下文拼接到 Prompt 中，调用 DeepSeek API 生成回答。为了支持更多任务，我使用 LangGraph 定义了状态、节点和条件路由，把问题分成文献问答、论文对比、实验记录查询和阅读笔记生成几类，不同意图会调用不同工具。

第三部分是评测和优化。我构造了测试问题集，每条问题包含 expected_source 和 expected_keywords。评测时统计检索是否命中预期文档、回答是否包含关键词、是否返回引用来源以及平均响应时间。根据失败样例，我会调整 chunk_size、chunk_overlap、TopK、Prompt 模板和检索阈值。

## 20.3 面试官可能追问

### 问：为什么要做引用来源？

答：因为 RAG 系统容易出现模型根据常识补全甚至编造内容的问题。返回引用来源可以让用户知道答案依据来自哪些文档，也方便排查检索是否命中正确内容。

### 问：chunk_size 怎么设置？

答：MVP 里我先设置为 800，overlap 设置为 120。太小会导致语义不完整，太大会导致检索片段不够精准。后续通过测试集评估 source_hit_rate 和 keyword_hit_rate，再调整参数。

### 问：LangGraph 和普通 Chain 有什么区别？

答：普通 Chain 更适合固定流程，比如单一 RAG 问答；LangGraph 更适合多步骤、有状态、需要条件路由的任务。我这个项目中不同问题可能走文献问答、论文对比、实验查询、笔记生成，所以用 LangGraph 更清晰。

### 问：怎么评估 RAG 效果？

答：我构建了测试问题集，每条问题标注 expected_source 和 expected_keywords，然后统计检索结果是否命中预期文档、回答是否包含关键词、是否返回引用来源和平均响应耗时。

### 问：如何降低幻觉？

答：主要做了四点：第一，Prompt 中约束必须依据参考资料回答；第二，检索结果不足时要求明确说明无法确认；第三，返回引用来源和相关片段；第四，通过测试集分析错误样例，优化检索和 Prompt。

# 21. 常见问题与排查

## 21.1 FAISS 加载报错

可能原因：

- 向量库还没有构建；
- 路径配置错误；
- embedding 模型与构建时不一致；
- `allow_dangerous_deserialization` 参数未设置。

解决：

```bash
python scripts/build_index.py
```

重新构建向量库。

## 21.2 检索结果不相关

可能原因：

- 文档内容太少；
- chunk_size 不合适；
- 用户问题和文档表达差异太大；
- Embedding 模型效果一般；
- TopK 太小。

解决：

- 增加文档内容；
- 调整 chunk_size；
- 调大 TopK；
- 优化文档标题和小节；
- 尝试中文 Embedding 模型；
- 增加 query rewrite。

## 21.3 LLM 回答编造

可能原因：

- Prompt 约束不够；
- context 太少；
- 检索结果不相关；
- temperature 太高。

解决：

- temperature 设置为 0.1 到 0.3；
- Prompt 中加入“资料不足请说明”；
- 返回引用来源；
- 检索不足时不调用生成或提示用户补充文档。

## 21.4 API Key 不生效

检查：

- `.env` 是否存在；
- 变量名是否一致；
- 是否重新启动服务；
- API Key 是否有效；
- base_url 是否正确。

## 21.5 PDF 解析乱码

解决：

- 换用文本型 PDF；
- 对扫描版 PDF 暂不支持；
- 后续可加入 OCR；
- 简历项目中不要写 OCR，除非真实实现。

## 21.6 实验记录查不到

可能原因：

- 查询方式太简单；
- 用户输入没有包含实验编号；
- JSON 数据字段不完整。

解决：

- 支持按 experiment_id 精确匹配；
- 支持 dataset / model / method 字段筛选；
- 对实验记录也构建向量索引。
