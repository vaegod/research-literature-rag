# 面试讲解稿

## 1 分钟版本

这个项目是一个科研文献智能问答与实验分析助手，主要解决论文阅读、实验记录查询和多文档对比效率低的问题。系统用 LangChain 完成文档加载、切分和向量化，用 FAISS 保存本地向量索引，用 FastAPI 提供接口，并用 LangGraph 编排 Agent 工作流。用户问题会被路由到文献问答、论文对比、实验查询或阅读笔记生成工具，答案会返回引用来源，便于检查依据。

## 3 分钟版本

项目主线分为三部分。

第一部分是 RAG 知识库构建。系统读取 PDF、Markdown 和 TXT 文档，按 chunk_size 和 overlap 切分，并保留 source、page、chunk_id 等 metadata。随后调用 OpenAI 兼容 Embedding 接口生成向量，写入本地 FAISS。

第二部分是问答与 Agent。普通 RAG 会先检索 TopK 片段，再拼接 Prompt 调用 Chat 模型生成答案。Agent 部分使用 LangGraph 定义状态、节点和条件路由，把问题分成文献问答、论文对比、实验记录查询、阅读笔记生成和普通问答。

第三部分是评测。项目维护了测试问题 CSV，每条问题包含 expected_source 和 expected_keywords。评测脚本统计来源命中率、关键词覆盖率和平均耗时，用于优化 TopK、chunk_size、overlap 和 Prompt。

## 常见追问

### 为什么要返回引用来源？

RAG 系统的一个风险是模型根据常识补全甚至编造内容。返回来源可以让用户知道答案依据来自哪些文档，也便于开发者排查检索是否命中正确片段。

### LangGraph 在这里解决什么问题？

普通 Chain 更适合固定流程，而这个项目的问题类型不同：有些需要检索文献，有些需要查询实验 JSON，有些需要生成阅读笔记。LangGraph 可以把意图分类、工具调用和回答生成组织成清晰的工作流。

### 如何评估效果？

MVP 阶段用 source_hit_rate、keyword_hit_rate 和 avg_latency。source_hit_rate 看是否命中预期文档，keyword_hit_rate 看答案是否覆盖关键概念，avg_latency 看响应速度。
