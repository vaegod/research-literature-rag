# 评测报告

本文档说明项目当前的评测设计、指标含义和后续改进方向。

## 1. 评测目标

当前评测不是为了证明模型能力，而是为了检查 RAG 系统是否完成三件事：

1. 检索是否命中正确资料。
2. 生成答案是否覆盖关键概念。
3. gold chunk 是否出现在 TopK 结果中。
4. 响应耗时是否在可接受范围内。

## 2. 评测数据

评测问题位于：

```text
data/eval/rag_eval_questions.csv
```

评测集中的 `expected_source` 应当对应当前 `data/raw_docs/` 中真实存在的样例文档，或特殊来源 `experiment_records.json`、`general_chat`。`tests/test_eval_data.py` 会在 Pytest 中检查这一点，避免评测题和样例知识库再次漂移。

每条记录包含：

| 字段 | 含义 |
| --- | --- |
| `id` | 样例编号 |
| `question` | 用户问题 |
| `intent` | 预期意图 |
| `expected_source` | 期望命中的来源 |
| `expected_keywords` | 答案中期望出现的关键词 |
| `expected_chunk_ids` | 可选 gold chunk，用于计算 Recall@K 和 MRR |

## 3. 指标

| 指标 | 计算方式 | 用途 |
| --- | --- | --- |
| `source_hit_rate` | 命中预期来源的样例比例 | 判断检索或工具调用是否找对资料 |
| `keyword_hit_rate` | 覆盖一半以上关键词的样例比例 | 判断生成答案是否包含关键概念 |
| `retrieval_recall_at_k` | 有 gold chunk 的样例中，TopK 是否至少命中一个 gold chunk | 判断检索链路是否把证据召回到上下文 |
| `mean_reciprocal_rank` | 第一个 gold chunk 排名的倒数均值 | 判断正确证据是否排在靠前位置 |
| `avg_latency` | 平均响应耗时 | 观察检索和模型调用开销 |
| `failed_cases` | 来源、关键词或 gold chunk 未通过的样例 | 用于迭代检索、Prompt 和切分参数 |

## 4. 运行方式

```powershell
cd 科研文献智能问答
.\.venv\Scripts\Activate.ps1
python scripts/run_eval.py
```

调试时建议先跑小样本：

```powershell
python scripts/run_eval.py --limit 5
```

评测结果会输出到：

```text
data/eval/eval_result.csv
```

该文件默认不提交到 GitHub，避免把临时答案和模型输出混入仓库。

## 5. 当前优化点

针对真实 PDF 检索中容易出现的“命中参考文献页、命中局部技术细节但没命中贡献段落”等问题，当前做了这些优化：

- 增加 chunk 的 `section` 元数据：abstract、introduction、method、experiments、conclusion、limitations、references、body。
- 对不同问题意图追加查询扩展词，提高英文论文的召回质量。
- 对贡献、方法、实验、局限等问题做 section 级别重排。
- 对 references 样式文本进行降权，减少回答被参考文献列表污染。

## 6. 局限

当前评测指标是轻量启发式指标，适合 MVP 快速迭代，但仍有局限：

- 关键词命中不能完全代表答案质量。
- 来源命中只看文件名或工具结果，不判断引用片段是否真正支持结论；gold chunk 可以缓解这个问题，但仍不能替代人工标注。
- 多跳问题和复杂对比问题需要更细粒度的人工评分。
- 同一问题的合理答案可能有多种表达，简单关键词会漏判。

## 7. 后续计划

- 扩大人工标注 gold chunks 覆盖面，用 Recall@K、MRR 持续衡量检索质量。
- 增加 LLM-as-a-judge，但要求 judge 输出引用依据，避免只看流畅度。
- 增加失败样例归因：检索失败、上下文不足、生成偏离、工具路由错误。
- 对真实论文 Demo 单独维护一组评测问题，覆盖贡献、方法、实验、局限和总结。
